Introduction to Version Control

Now that you’ve finished installing all the necessary software, it’s time to integrate Git into your workflow! As stated before, Git is a version control software designed to track changes in files. This feature will be extremely helpful as your research project evolves over time.

You can imagine that there may be certain points in your project where you are investigating several different directions and it becomes hard to keep track of what has already happened. You may even want to undo or redo some steps as you get feedback, like the comic above illustrates.

With Git, all your project history is cleanly recorded, allowing you easy access to any point in your history without cluttering your computer with all of the file iterations that precede the final version. Additionally, Git allows collaborators to jump in and safely contribute to your project in an organized manner, making it a highly versatile and valuable software to use.

Git vs. GitHub

Git and GitHub have similar names, but it’s important to note the differences in their function!

Git:

  • is an open-source distributed version control software, designed to manage the versioning of files
  • operates locally on your computer, and can be enabled on a specific folder/directory to version the files inside
    • this tracked folder is called a repository
  • includes features such as creating repositories, committing changes, branching and merging code
  • primarily used via command line code, but there are also many GUIs like the RStudio Git feature

GitHub:

  • is an online platform/website built around Git
  • provides a centralized location to host and share Git repos in the cloud
  • offers additional features on top of Git, such as a web-based interface, issue tracking, project management tools, pull requests, and code review

Workflow Overview

The typical Git workflow can be daunting for a beginner to figure out how all these moving parts work together, so here is a brief overview, simplified as much as possible.

Suppose we have a project folder on our local computer, my_project that contains our scripts, data_download.R, data_prep.R and data_analysis.R like so:

my_project/            # our working directory
├── data_download.R    # first working file 
│    
├── data_prep.R        # second working file 
│
└── data_analysis.R    # third working file

In this example, my_project is our working directory and the R scripts are our local working files, which are the files that we edit as we work.

Now suppose we want to start using Git to track the changes in the my_project folder. To do this, the command is called initializing my_project as a Git repository. By initializing a folder as a Git repo, we are telling Git to start tracking all the contents inside this folder.

This initialization step will create a hidden .git folder, which is the actual Git repository that will have the history of our files. This Git repo is also referred to as our local repo since it is on our local computer.

my_project/            # our working directory
├── data_download.R    # first working file 
│    
├── data_prep.R        # second working file 
│
├── data_analysis.R    # third working file
│
└── .git/              # our local repo
    └── ...
Note

By the way, the way to tell the difference between a normal folder and a folder tracked by Git is precisely the presence of that hidden .git folder! If you’re unsure whether a folder has been initialized by Git already, you can change your computer’s settings to enable seeing hidden folders and check if the folder has a .git folder inside.

Then as we edit our working files, we can use Git commands to save versions of our files locally. Once we’re satisfied, we can finally send our edits to GitHub, a popular destination to host remote repos (repositories that live in the cloud).

The diagram below illustrates the Git commands needed to carry out the basic workflow.

In the diagram, rabbits are moving back and forth between 2 islands: an island with the remote repo (representing the cloud) and an island with the working directory + local repo (representing the local computer).

Starting in the bottom right-hand corner, the rabbits in the working directory move file boxes to the staging area with the add command. Then they move the file boxes from the staging area to the local repo with the commit command.

Finally, the rabbits get in a boat to move the boxes from the local repo to the remote repo with the push command. To move boxes from the remote repo back to the local repo, they use the pull command. There is also a dotted line in the center to signify that it’s best practice to pull before pushing.

Although it’s common to initialize a Git repo on your local computer first and then send it to GitHub, you can also go the opposite way! Some people may find it easier to start the remote repo on GitHub first and then clone it to their local computer. Either method will work and they’re both valid. This tutorial will focus on the latter method.

Git Vocabulary Summary

To summarize, here are all the basic Git vocabulary terms covered thus far and their respective commands:

Term Git Command Definition
Initialize git init initializes a brand new local Git repo (*)
Add/Stage git add [file] marks the local modified file(s) going into the next commit by putting them in the staging area
Commit git commit -m "[my commit message]" captures a snapshot of currently staged local changes with a message describing what exactly has been changed; this snapshot goes into our local repo
Pull git pull grabs changes from a remote repo and merges them into your local working file(s)
Push git push sends local changes to a remote repo
Clone git clone [repo] creates a local copy of a remote repo (*)

(*) You can start your workflow either by initializing an already existing local folder as a Git repo and then pushing to GitHub, OR creating a remote repo on GitHub and then cloning it to your local computer!

Don’t worry about memorizing the exact commands, we can utilize the GUIs available in RStudio, VS Code, or GitHub Desktop to carry out the same operations with simple button clicks.

If you would like to see the full list of commands, visit the Git documentation.
