Git Workflow

Warning

Ensure you have completed the steps in Installation before attempting to do the tutorial on this page!

Introduction to Version Control

Now that you’ve finished installing all the necessary software, it’s time to integrate Git into your workflow! As stated before, Git is a version control software designed to track changes in files. This feature will be extremely helpful as your research project evolves over time.

You can imagine that there may be certain points in your project where you are investigating several different directions and it becomes hard to keep track of what has already happened. You may even want to undo or redo some steps as you get feedback, like the comic above illustrates.

With Git, all your project history is cleanly recorded, allowing you easy access to any point in your history without cluttering your computer with all of the file iterations that precede the final version. Additionally, Git allows collaborators to jump in and safely contribute to your project in an organized manner, making it a highly versatile and valuable software to use.

Git vs. GitHub

Git and GitHub have similar names, but it’s important to note the differences in their function!

Git:

  • is an open-source distributed version control software, designed to manage the versioning of files
  • operates locally on your computer, and can be enabled on a specific folder/directory to version the files inside
    • this tracked folder is called a repository or “repo” for short
  • includes features such as creating repositories, committing changes, branching and merging code
  • primarily used via command line code, but there are also many GUIs like the RStudio Git feature

GitHub:

  • is an online platform/website built around Git
  • provides a centralized location to host and share Git repos in the cloud
  • offers additional features on top of Git, such as a web-based interface, issue tracking, project management tools, pull requests, and code review

Workflow Overview

The typical Git workflow can be daunting for a beginner to figure out how all these moving parts work together, so here is a brief overview, simplified as much as possible.

Suppose we have a project folder on our local computer, my_project that contains our scripts, data_download.R, data_prep.R and data_analysis.R like so:

my_project/            # Our working directory
├── data_download.R    # First working file 
│    
├── data_prep.R        # Second working file 
│
└── data_analysis.R    # Third working file

In this example, my_project is our working directory and the R scripts are our local working files, which are the files that we edit as we work.

Now suppose we want to start using Git to track the changes in the my_project folder. To do this, the command is called initializing my_project as a Git repository. By initializing a folder as a Git repo, we are telling Git to start tracking all the contents inside this folder.

This initialization step will create a hidden .git folder, which is the actual Git repository that will have the history of our files. This Git repo is also referred to as our local repo since it is on our local computer.

my_project/            # Our working directory
├── data_download.R    # First working file 
│    
├── data_prep.R        # Second working file 
│
├── data_analysis.R    # Third working file
│
└── .git/              # Our local repo that will have 
    └── ...            # the saved history of our files
Note

By the way, the way to tell the difference between a normal folder and a folder tracked by Git is precisely the presence of that hidden .git folder! If you’re unsure whether a folder has been initialized by Git already, you can change your computer’s settings to enable seeing hidden folders and check if the folder has a .git folder inside.

Then as we edit our working files, we can use Git commands to save versions of our files locally. Once we’re satisfied, we can finally send our edits to GitHub, a popular destination to host remote repos (repositories that live in the cloud).

The diagram below illustrates the Git commands needed to carry out the basic workflow.

In the diagram, rabbits are moving back and forth between 2 islands: an island with the remote repo (representing the cloud) and an island with the working directory + local repo (representing the local computer).

Starting in the bottom right-hand corner, the rabbits in the working directory move file boxes to the staging area with the add command. Then they move the file boxes from the staging area to the local repo with the commit command.

Finally, the rabbits get in a boat to move the boxes from the local repo to the remote repo with the push command. To move boxes from the remote repo back to the local repo, they use the pull command. There is also a dotted line in the center to signify that it’s best practice to pull before pushing.

Although it’s common to initialize a Git repo on your local computer first and then send it to GitHub, you can also go the opposite way! Some people may find it easier to start the remote repo on GitHub first and then clone it to their local computer. Either method will work and they’re both valid. This tutorial will focus on the latter method.

Git Vocabulary Summary

To summarize, here are all the basic Git vocabulary terms covered thus far and their respective commands:

Term Git Command Definition
Initialize git init initializes a brand new local Git repo (*)
Add/Stage git add [file] marks the local modified file(s) going into the next commit by putting them in the staging area
Commit git commit -m "[my commit message]" captures a snapshot of currently staged local changes with a message describing what exactly has been changed; this snapshot goes into our local repo
Pull git pull grabs changes from a remote repo and merges them into your local working file(s)
Push git push sends local changes to a remote repo
Clone git clone [repo] creates a local copy of a remote repo (*)

(*) You can start your workflow either by initializing an already existing local folder as a Git repo and then pushing to GitHub, OR creating a remote repo on GitHub and then cloning it to your local computer!

If you would like to see the full list of commands, visit the Git documentation.

Don’t worry about memorizing the exact commands, we can utilize the GUIs available in RStudio, VS Code, or GitHub Desktop to carry out the same operations with simple button clicks.

Practice

Now let’s practice the Git workflow by first creating a remote repo on GitHub and then cloning it to our local computer!

Creating a repo on GitHub

Log into GitHub. Once you’re logged in, click on your account icon in the top right corner and go to “Repositories”.

Click on “New” to create your new repository.

Fill out your practice repo name, write a short description to summarize what this repo contains, set the visibility to “Public”, add a README, and add a gitignore template that corresponds to the programming language you will use for this exercise. There are tons of gitignore template options for various languages, like R, Python, Java, C++, etc.

Note

A gitignore is a file that does exactly what it says— it tells Git to ignore certain files! While working on your project, there may be certain files or file types you don’t wish to track the changes of, in which case, you can list them inside the gitignore. For example, if your code exports a lot of png files that don’t need to be tracked by Git, you can type “*.png” in the gitignore to instruct Git to ignore all files with the “.png” extension.

We won’t add a license to this practice repo but for actual research projects in MDEL, the most common license type is the MIT License. Check out the GitHub documentation on licenses to learn more.

Once the form is filled out, click “Create repository” and wait for the process to finish. Your new practice repo may look like this:

The landing page shows the top folders/files in your repo as well as some information on when those files were modified. Right now, the repo only has a gitignore and README file that were created just a moment ago.

If you’re curious about the function of a gitignore, click on the file and see what file types are inside for your specified coding language. For example, here are the file types for the R gitignore template:

Miscellaneous files like .Rhistory and .Rdata are listed inside. If you’re a regular R user, you may recognize these file types as the files that get frequently get created by R during your session. These files don’t impact your actual work so it makes sense for Git to ignore them.

Cloning to your local computer

To get a copy of this remote repo to our local computer, we will need to clone it. The Git command to do so is git clone, but IDEs like RStudio and VS Code provide easy-to-use interfaces that allow you to do the same operation with button clicks.

If you don’t use RStudio or VS Code, feel free to try out GitHub Desktop instead.

  • RStudio users
  • VS Code users
  • GitHub Desktop

Go to the landing page of your practice repo on GitHub. Click on the “Code” then copy the link under “HTTPS” in the dropdown menu.

Switch to RStudio. Click on the top right corner that has an icon of a box with an R on it.

In the resulting dropdown menu, click “New Project”.

Then select “Version Control”.

And then “Git”.

In the following page, paste the link you copied from GitHub. For the local copy of your repo, you can name it a different name from your remote repo if you wish, but it’s generally recommended to keep the name the same. You can also choose where to clone the repo on your computer. If you wish to easily remember what folders are Git repos versus normal folders on your computer, you can create a folder called “GitHub” where all your local Git repos can live.

Otherwise, once you’re satisfied, click “Create Project”.

Now we have cloned the repo to our local computer! Notice in the top right corner that we’re now working inside the practice repo, and that we have local copies of the gitignore and README.

Pushing a local file to GitHub

Let’s practice a simple Git workflow where we create a new file on our local computer and send it to GitHub.

  • RStudio users
  • VS Code users
  • GitHub Desktop

Start by creating a new practice script in RStudio. Feel free to write any code in the script as this is just an example.

Once you’re satisfied, click on the “Git” tab and then click on either the “Diff” or “Commit” tabs. These two tabs will lead to the same place, so don’t worry about clicking on either one.

A new window should pop up. Click on your practice script and notice how there are green lines highlighted in the window. The green lines signify the new additions to this file since our last commit. Since this file has never been tracked by Git before, all the lines are highlighted in green and there is a yellow question mark icon next to the script name.

To explain more about what these icons mean, they represent the status of each file in your repo. The yellow question mark signifies a file that is not yet tracked by Git. Here are some other icons you might see working in RStudio:

If you think back to the Git Workflow Overview, in order to send this practice script to GitHub, the first step is to add it to the staging area. The Git command for this is git add, but thankfully, we can do the equivalent operation by checking the checkbox to the left of the script name.

Notice how the yellow question mark icon turned into a green A icon, which represents a new file that is now marked for tracking.

You can imagine that in a regular project, you may want to stage several files in a single commit by checking multiple checkboxes, but we will only focus on staging this one file for this exercise.

After staging the file, we will need to commit it, which will capture a snapshot of any currently staged local changes. The corresponding Git command for this is git commit -m "[my commit message]", but we can use the RStudio GUI instead. Write a commit message to describe exactly what we’ve changed in the right-hand panel, and then click “Commit”.

Once you click “Commit”, you might see a new window pop-up like this:

Close the pop-up window to get back to your main workspace and you will see a message that says, “Your branch is ahead of ‘origin/main’ by 1 commit.” This means that we have a local commit that the remote repo on GitHub does not have yet. In a regular project, you would probably have several commits before you’re ready to push, but for the purposes of this exercise, we will send this one commit to GitHub.

To send this local commit to GitHub, we will want to pull (git pull) first and then push (git push). Pulling will fetch the latest changes on GitHub and merge them with our local files. It’s best practice to pull often and to always pull BEFORE pushing. It makes sense to grab the latest version of the files on GitHub and then make our local edits on top of that. After we make our local edits, then we can push them back to GitHub. This will create a clean history of your repo.

This habit will really save you whenever you’re working in a repo with other collaborators, who may make edits when you’re away from your computer. By pulling frequently, you can avoid merge conflicts. You can always pull at any point in your workflow, so don’t hesitate to make it a habit.

With that said, click on the “Pull” arrow to fetch and merge the latest changes from GitHub.

It will say that we’re already up to date because we haven’t done anything to our GitHub repo since we’ve last touched it.

Close that window and click “Push” arrow to send our local changes to GitHub.

If you see a pop-up window that looks kind of like this, you’ve successfully pushed to GitHub!

You can confirm by navigating back to your GitHub repo. Check to see that you now have a second commit where you created your practice script. If you click on where it says “2 Commits”, you can view the history of all your commits.

Click on your latest commit.

You should be able to see the green lines again showing the new additions to this file.

Congrats! You’ve now completed a Git workflow where you created a repo on GitHub, cloned it to your local computer, created a local file, and pushed it back to GitHub! You can rinse and repeat this workflow during the course of your projects. Remember, after you have your remote and local repos set up, the correct order is to make your local edits –> stage your local changes –> commit them with descriptive commit messages –> pull –> push to your remote repo.
